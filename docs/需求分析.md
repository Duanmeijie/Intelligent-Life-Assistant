# 智能生活助手 — 需求分析文档

## 1. 项目概述

### 1.1 项目背景

智能生活助手（Intelligent Life Assistant）是一款面向个人用户的**时间管理与习惯追踪平台**。在现代快节奏的生活中，人们面临着时间碎片化、目标难以坚持、效率难以量化等痛点。本项目旨在通过数字化的手段，帮助用户科学地管理日程、培养良好习惯、追踪每日打卡情况，并借助数据可视化与 AI 分析能力，为用户提供个性化的效率提升建议。

### 1.2 项目目标

- 提供一个集日程管理、习惯养成、打卡追踪于一体的个人效率工具
- 通过社交社区功能，营造积极向上的习惯养成氛围
- 利用数据可视化（ECharts）直观展示用户行为数据
- 引入 AI 推荐引擎，为用户的习惯培养和日程安排提供智能建议

### 1.3 目标用户

- **普通用户**：希望提升个人时间管理能力和养成良好习惯的职场人士、学生群体
- **管理员**：负责平台运营、用户管理与系统监控的后台人员

### 1.4 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | Vue 3 + Vite |
| 状态管理 | Pinia |
| 路由管理 | Vue Router 4 |
| UI 组件库 | Element Plus |
| 图标库 | @element-plus/icons-vue |
| 数据可视化 | ECharts 5 + vue-echarts |
| HTTP 请求 | Axios |
| 日期处理 | dayjs |
| 后端框架 | Node.js + Express |
| 数据库 | MySQL 8.0 |
| 缓存 | Redis |
| 认证方式 | JWT（JSON Web Token） |
| 密码加密 | bcryptjs |
| 数据校验 | express-validator |
| 日志记录 | morgan |

## 2. 功能需求

### 2.1 用户管理

#### 2.1.1 用户注册

- 用户可输入**用户名**（3-20 位，仅支持字母、数字和下划线）、**邮箱**（合法格式）、**密码**（8-20 位，须包含大小写字母和数字）进行注册
- 系统校验用户名和邮箱的唯一性，若已存在则返回 409 冲突
- 注册成功后，密码经 bcrypt（salt rounds = 10）加密存储
- 注册成功后自动签发 JWT 令牌（有效期为 7 天），返回用户基本信息与令牌

#### 2.1.2 用户登录

- 用户通过**用户名 + 密码**进行登录
- 校验账户状态，若账户被禁用（status = 0）则拒绝登录
- 密码验证通过后签发 JWT 令牌
- 返回用户完整信息（id、用户名、邮箱、昵称、头像、个人简介）与令牌

#### 2.1.3 JWT 认证

- 前端请求携带 `Authorization: Bearer <token>` 头部
- 后端统一在中间件层进行令牌校验
- 令牌过期自动返回 401，前端清除本地存储并跳转登录页
- 支持可选的认证中间件（optionalAuth），用于不强制登录的场景

#### 2.1.4 个人资料管理

- 用户可查看个人资料
- 支持修改**昵称**（最长 50 字符）、**个人简介**（最长 255 字符）、**头像 URL**

#### 2.1.5 密码修改

- 用户需提供**原密码**和**新密码**进行修改
- 新密码同样需要满足复杂度要求（8-20 位，含大小写字母和数字）
- 原密码校验不通过则返回 400 错误

### 2.2 日程管理

#### 2.2.1 功能描述

用户可对自己的时间安排进行精细化管理，支持日程的增删改查以及状态流转。

#### 2.2.2 分类体系

| 分类 | 标识 | 说明 |
|------|------|------|
| 工作 | work | 工作任务、会议、项目等 |
| 学习 | study | 课程、阅读、技能提升等 |
| 生活 | life | 家庭、休闲、社交等 |
| 运动 | sports | 健身、跑步、瑜伽等 |
| 其他 | other | 其他未分类日程 |

#### 2.2.3 状态流转

| 状态 | 标识 | 说明 |
|------|------|------|
| 待开始 | pending | 日程已创建但尚未开始 |
| 进行中 | in_progress | 日程正在进行中 |
| 已完成 | completed | 日程已完成 |
| 已取消 | cancelled | 日程已被取消 |

状态支持任意方向流转，用户可根据实际情况灵活调整。

#### 2.2.4 功能列表

- **创建日程**：填写标题（必填，最长 100 字符）、描述（可选，最长 500 字符）、分类（可选，默认 other）、开始时间（必填，ISO8601 格式）、结束时间（必填，ISO8601 格式）
- **编辑日程**：支持修改除 user_id 外的所有字段
- **删除日程**：软删除，仅创建者可删除
- **查询日程**：支持按日期范围、分类、状态进行过滤，支持分页
- **状态更新**：独立的 PUT 接口支持仅更新日程状态

### 2.3 习惯追踪

#### 2.3.1 功能描述

用户可创建并维护个人习惯，通过持续的打卡行为培养自律能力。

#### 2.3.2 频率类型

| 频率 | 标识 | 说明 |
|------|------|------|
| 每日 | daily | 每天执行一次 |
| 每周 | weekly | 每周执行一次 |
| 每月 | monthly | 每月执行一次 |
| 自定义 | custom | 用户自定义频率 |

#### 2.3.3 功能列表

- **创建习惯**：填写习惯名称（必填，最长 100 字符）、描述（可选，最长 500 字符）、频率（可选，默认 daily）、分类（可选）、颜色标识（可选，默认 #409EFF，16 进制格式）、提醒时间（可选）
- **编辑习惯**：支持部分更新，可修改任意字段
- **删除习惯**：仅创建者可删除，关联的打卡记录受外键级联删除
- **查询习惯**：获取用户所有习惯列表，支持活跃状态筛选
- **活跃/暂停**：通过 is_active 字段控制习惯的启用/暂停状态

### 2.4 打卡签到

#### 2.4.1 功能描述

用户针对已创建的习惯进行每日打卡，记录执行情况。

#### 2.4.2 核心功能

- **每日打卡**：用户选择一个习惯，记录打卡日期（默认当天）和备注（可选）
- **重复打卡校验**：同一习惯同一天不允许重复打卡（通过 unique key: habit_id + checkin_date 保证）
- **打卡记录查询**：支持按习惯 ID、日期范围筛选，支持分页
- **打卡统计**：统计各习惯的打卡总次数、完成率、连续打卡天数
- **连续打卡计算**：分别计算当前连续天数和历史最长连续天数
- **撤销打卡**：用户可删除自己的打卡记录

#### 2.4.3 统计指标

- 总打卡次数
- 完成率（打卡次数 / 统计周期天数 × 100%）
- 当前连续打卡天数
- 历史最长连续打卡天数

### 2.5 社交社区

#### 2.5.1 功能描述

用户可在社区中发布动态、分享习惯养成心得、进行互动交流。

#### 2.5.2 动态管理

- **发布动态**：填写内容（必填，最长 2000 字符），可附带图片列表（JSON 数组格式）
- **删除动态**：仅发布者可删除
- **动态列表**：分页查询，返回用户信息（用户名、昵称、头像）、点赞数、评论数
- **动态状态**：支持正常/隐藏两种状态

#### 2.5.3 点赞功能

- **切换点赞**：同一用户对同一动态只能点赞一次，再次点击取消点赞
- 点赞/取消点赞时自动更新 social_posts 表中的 likes_count 计数
- 数据一致性通过 unique key（user_id + post_id）保证

#### 2.5.4 评论功能

- **添加评论**：对动态发表评论（内容最长 500 字符），自动增加动态的评论计数
- **评论列表**：查询指定动态的所有评论，返回评论用户信息
- **回复功能**：通过 parent_id 字段支持评论的嵌套回复（预留）

### 2.6 AI 智能推荐

#### 2.6.1 习惯推荐

基于用户已有习惯和打卡数据分析，提供个性化的习惯推荐：

- **新用户场景**：尚无任何习惯时，推荐 3 个不同分类的通用习惯
- **老用户场景**：
  - 分析过去 N 天（默认 30 天）的打卡数据
  - 识别完成率低于 50% 的习惯，推荐同类辅助习惯
  - 识别整体完成率低于 30% 的弱势分类，推荐该分类的习惯
  - 识别用户尚未涉及的分类（如缺少"运动"类），推荐新分类的习惯
- 最多返回 5 条推荐结果，每条附带推荐理由和难度评估

#### 2.6.2 效率分析

对用户过去一段时间的行为进行全面分析：

- **总览统计**：习惯总数、总打卡次数、整体完成率、周均打卡次数、日程完成率
- **习惯分析**：每个习惯的打卡天数、完成率、评级（优秀/良好/一般/需要加强）
- **每日趋势**：逐日打卡数量变化
- **日程统计**：各分类的日程总数和完成数
- **强弱项识别**：自动识别表现最好和最差的分类
- **改进建议**：根据完成率给出分级的文字建议

#### 2.6.3 日程建议

分析用户的日程安排模式，提供优化建议：

- **模式识别**：自动分析用户在什么时间段安排什么类型的日程
- **空闲时段发现**：识别一天中用户没有安排的时间段
- **分类缺失检测**：检测用户是否忽略了某个分类的日程安排
- **默认建议**：新用户可获取基于最佳实践的通用日程模版
- 每个建议附带推荐理由，帮助用户理解建议依据

### 2.7 数据可视化

#### 2.7.1 习惯统计看板

- **柱状图**：展示各习惯的完成率（百分比），使用渐变色渲染，数据标签显式标注
- **饼图**：展示习惯的分类分布情况，使用环形图样式

#### 2.7.2 打卡热力图

- 类似 GitHub 贡献图的年度打卡日历
- 以天为单位展示用户的打卡密度
- 支持年份切换查看

#### 2.7.3 日程统计看板

- **饼图**：展示日程的分类分布
- **柱状图**：展示每周日程数量变化趋势

#### 2.7.4 首页仪表盘

- 展示关键指标卡片：今日日程数、活跃习惯数、今日打卡率、连续打卡天数
- 展示今日日程列表（按时间排序）
- 展示习惯列表并支持一键打卡操作
- 根据当前时间自动切换问候语（如"早上好""下午好"等）

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | ≤ 2 秒（首屏加载） |
| API 响应时间 | ≤ 500 毫秒（95% 的请求） |
| API 超时设置 | 15 秒（前端 Axios 配置） |
| 数据库连接池 | 最大 20 个连接 |
| 并发用户数 | 支持 100+ 用户同时在线 |

### 3.2 安全需求

| 需求 | 措施 |
|------|------|
| 认证安全 | JWT 令牌认证，令牌有效期 7 天 |
| 密码安全 | bcryptjs 加密（salt rounds = 10） |
| 密码复杂度 | 须包含大小写字母和数字，长度 8-20 |
| 接口鉴权 | 关键接口统一经过 auth 中间件鉴权 |
| 数据隔离 | 用户只能操作自己的数据（user_id 校验） |
| SQL 注入防护 | 使用参数化查询（mysql2/prepared statements） |
| 请求校验 | express-validator 对输入做合法性校验 |

### 3.3 响应式设计

- 适配桌面端（> 768px）和移动端（≤ 768px）
- 桌面端：左侧固定侧边栏（220px），支持折叠（64px）
- 移动端：自动切换为抽屉式导航
- 栅格系统自适应布局（el-row/el-col）
- 统计卡片在移动端自动调整为两列显示
- 图表容器自适应窗口 resize

### 3.4 可扩展性

- **模块化架构**：前后端分离，各功能模块独立解耦
- **后端分层**：routes → controllers → models 三层架构
- **前端组件化**：公共组件独立维护（AppLayout、CalendarHeatmap、PostCard 等）
- **API 版本化**：以 `/api` 为前缀，预留版本扩展空间
- **中间件机制**：认证、日志、错误处理、数据校验均为独立中间件
- **状态管理分离**：Pinia store 按功能域拆分（auth、habit、schedule）

### 3.5 可用性需求

- 服务端启动时自动检测数据库连接状态
- 统一的错误处理中间件，避免未捕获异常
- 前端请求/响应拦截器统一处理错误提示和 401 重定向
- 提供健康检查接口（`/api/health`）

## 4. 业务流程

### 4.1 用户注册与登录流程

```
[用户] → 访问注册页面 → 填写用户名/邮箱/密码
    → 前端格式校验（长度、复杂度、格式）
    → [API: POST /api/auth/register]
        → 后端 express-validator 二次校验
        → 检查用户名唯一性 → 重名则返回 409
        → 检查邮箱唯一性 → 已存在则返回 409
        → bcrypt.hash(password, 10) 加密
        → 写入 users 表
        → jwt.sign() 签发令牌
    ← 返回 201 { token, user }
    → 前端 Pinia store 存储 token 和用户信息
    → 跳转至首页仪表盘

[用户] → 访问登录页面 → 输入用户名/密码
    → [API: POST /api/auth/login]
        → 根据用户名查询用户
        → 校验用户状态（是否被禁用）
        → bcrypt.compare() 验证密码
        → jwt.sign() 签发令牌
    ← 返回 { token, user }
    → 前端存储 token 到 localStorage
    → 页面跳转至首页（或之前试图访问的重定向页面）
```

### 4.2 日程创建与管理流程

```
[用户] → 进入日程管理页面
    → 页面自动加载日程列表（分页）
    → 支持按日期范围、分类、状态过滤

创建日程：
    → 点击"新建日程"按钮 → 弹出表单
    → 填写标题、描述、分类、起止时间
    → 前端校验 → [API: POST /api/schedules]
        → 后端校验参数合法性
        → 写入 schedules 表
    ← 返回 201 与创建的日程对象
    → 刷新列表

状态变更：
    → 用户点击状态标签或操作按钮
    → [API: PUT /api/schedules/:id/status]
        → 校验日程归属权
        → 更新 status 字段
    ← 返回更新后的日程

编辑/删除流程类似，均需校验操作权限。
```

### 4.3 习惯创建与每日打卡流程

```
[用户] → 进入习惯管理页面
    → 查看已有习惯列表

创建习惯：
    → 填写习惯名称、分类、频率、颜色、提醒时间
    → [API: POST /api/habits]
        → 校验参数合法性
        → 写入 habits 表
    ← 返回 201 与创建的习惯

每日打卡：
    → 用户可通过首页仪表盘或打卡页面操作
    → 选择需要打卡的习惯
    → [API: POST /api/checkins]
        → 检查 habit_id + checkin_date 联合唯一约束
        → 已打卡 → 返回 409 重复打卡提示
        → 未打卡 → 写入 checkins 表
    ← 返回 201 "打卡成功"
    → 前端更新打卡状态（checkedIn = true）

撤销打卡：
    → [API: DELETE /api/checkins/:id]
        → 校验记录归属权
        → 软删除打卡记录
```

### 4.4 社交互动流程

```
[用户] → 进入社区页面
    → 浏览动态列表（分页展示）

发布动态：
    → 编辑内容（支持图片）
    → [API: POST /api/social]
        → 校验内容非空且不超过 2000 字符
        → 写入 social_posts 表（images 以 JSON 存储）
    ← 返回 201

互动：
    → 浏览动态时，可点击点赞按钮
        → [API: POST /api/social/:id/like]
            → 检查是否已点赞
            → 未点赞 → SocialLike.create() + 递增 likes_count
            → 已点赞 → SocialLike.delete() + 递减 likes_count
        ← 返回点赞/取消成功

    → 浏览动态时，可点击评论按钮
        → 输入评论内容
        → [API: POST /api/social/:id/comment]
            → 校验内容非空且不超过 500 字符
            → 写入 social_comments 表 + 递增 comments_count
        ← 返回 201 与评论信息（含用户信息）
```

### 4.5 AI 分析与推荐流程

```
习惯推荐流程：
    [用户] → 进入习惯推荐页面
    → [API: GET /api/ai/recommend-habits?days=30]
        → 查询用户的习惯列表和过去 N 天的打卡数据
        → 未找到习惯 → 返回通用分类推荐
        → 已找到习惯 →
            1. 筛选完成率 < 50% 的习惯 → 推荐同类习惯
            2. 识别整体完成率 < 30% 的分类 → 推荐该分类
            3. 发现缺失的分类 → 推荐新分类习惯
        → 去重、限制最多 5 条
    ← 返回推荐列表（含名称、分类、颜色、理由、难度）

效率分析流程：
    [用户] → 进入数据分析 → 效率分析标签页
    → [API: GET /api/ai/analysis?days=30]
        → 查询习惯 + 打卡 + 日程数据
        → 计算各习惯完成率
        → 生成每日趋势数据
        → 分析强弱分类
        → 生成文字建议
    ← 返回完整分析报告

日程建议流程：
    [用户] → 进入相关页面
    → [API: GET /api/ai/schedule-suggest]
        → 分析用户日程时间分布模式
        → 找出空闲时段
        → 检测缺失分类
        → 无数据时返回通用日程模版
    ← 返回建议列表与模式分析
```

## 5. 角色用例

### 5.1 普通用户

| 用例编号 | 用例名称 | 前置条件 | 后置条件 |
|----------|----------|----------|----------|

## 6. 数据库设计

### 6.1 整体设计规范

- **字符集**：utf8mb4（支持完整 Unicode，包括 emoji）
- **排序规则**：utf8mb4_unicode_ci
- **存储引擎**：InnoDB（支持事务、外键、行级锁）
- **命名规范**：表名使用小写复数形式，字段使用小写蛇形命名
- **时间字段**：统一使用 DATETIME 类型
- **软删除**：物理删除（外键 ON DELETE CASCADE）

### 7.4 数据流说明

1. **用户请求流程**：Vue 页面 → Pinia Store → Axios API 请求 → 请求拦截器注入 Token → Express 路由 → 认证中间件 → 控制器 → 数据模型 → MySQL/Redis → 响应 → 控制器格式化 → 响应拦截器处理 → 页面更新

2. **认证数据流**：登录/注册 → JWT 签发 → 前端 localStorage 存储 → 每次请求通过拦截器携带 → 后端 auth 中间件验证 → decoded 注入 req.user

3. **缓存策略**：Redis 作为缓存层（当前为可选组件，通过 `config/redis.js` 配置连接），可用于会话缓存、打卡排行榜等高频读取场景
